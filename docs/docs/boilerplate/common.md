大部分模板都会使用相同的技术栈，不同的是针对一些特定的场景会有一些特定的逻辑。这里整理一些通用的逻辑供参考。

## 目录名称约定

| 目录 | 作用 | 备注 |
|------|-----|--------|
| `constants` | 常量相关 | 存储一些常量，如接口地址、错误码等 |
| `hoc` | react hoc组件 | 高阶组件，用于封装一些通用的逻辑 |
| `components` | 页面组件 | 页面组件，用于封装一些通用的组件 |
| `containers` | 容器组件 | 容器组件，用于封装一些通用的容器组件 |
| `utils` | 幂等的函数 | 纯函数，每次输入保证有相同输出的函数，如一些工具函数 |
| `helpers` | 非幂等的函数 | 非函数，每次输入不保证相同输出的函数或产生了某种副作用的函数 |
| `hooks` | hooks | 自定义 hooks，用于封装一些通用的逻辑 |
| `images` | 图片资源 | 存储一些图片资源 |
| `styles` | css | 存储一些 css 样式 |
| `lang` | 国际化语言文件 | 存储国际化语言文件 |
| `modules` | 页面模块 | 通常一个页面为一个模块 |

### 如何区分`components`和`containers`

* `components`：偏向更小一些的组件，或者更通用的组件，比如一个按钮、一个特定功能的组件等。
* `containers`：容器组件，是一大段业务组件的集合，比如一个页面、一个模块等，比如tabs页面，一个tab属于一个`container`。

### 如何区分`utils`和`helpers`

没有副作用的函数属于`utils`，比如一些工具函数，一些数据的转换、日期的转换等等。

有副作用的函数就属于`helpers`，比如一些业务逻辑，比如一些数据的获取、处理、展示等。

## 固定功能目录约定

### lang目录

在`esboot`中，**所有多语言文件都放在`src/lang`目录下**，并且所有的语言文件都放在一个文件夹下，比如`zh-CN.json`、`zh-TW.json`、`en-US.json`。可以通过`xx.xx.xx`来区分不同的层级。比如：

```json
{
  "global": {
    "project": "测试项目名称",
    "boundary_error_title": "哎呀，出错了！",
    "apologize": "抱歉，应用程序遇到了一些问题。",
    "retry": "重试"
  }
}
```

**📚 为什么要这样做？**

- 使用`json`来管理多语言，可以避免`ts`的灵活性带来的问题，特别是后期增加语言种类的时候，可以避免到处导入导致的问题。
- `ESBoot`的`lint`规则会检查`lang`目录下的语言文件，确保语言key的合理、规范。
- 统一管理多语言文件，可以避免重复字段的问题，也可以避免遗漏字段的问题。

### styles目录

在`ESBoot`中，**所有全局css文件都应该放在`src/styles`目录下**，`ESBoot`默认会开启`scss module`
